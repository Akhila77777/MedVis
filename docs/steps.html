Build Steps: The Needle-Planning Slicer Module

A hands-on build log for Module 3 of the learning guide — roadmap tasks #4 (fiducials + trajectory) and #5 (vessel clearance check).

How to use this page: work through the steps in order. Each one has four parts — What (the concrete action), Why (the reasoning, so you're not just following clicks), How (menu paths / code), and Underlying concept (the theory — this is what you'd explain in an interview). Yellow boxes call out specific "they might ask you this" angles. This file will grow with new sections as later tasks (Capella model, requirements, risk analysis, MONAI) get their own build steps.

Before you start

You'll need, inside 3D Slicer:

  • A volume loaded (the MRHead sample or the CT Thorax/Abdomen you already loaded both work for Steps 1–3).
  • For Step 4 (vessel clearance), any segmentation will do for now — it doesn't need to be an anatomically correct vessel yet. Use the Segment Editor module, create a new segment, and use the "Threshold" effect to segment something solid (e.g. bone in the CT, or the skull in MRHead). The goal is to have some closed 3D structure to test the intersection logic against. You'll swap in a real liver/vessel segmentation once the Decathlon dataset (Task #2) is downloaded.
This module covers two roadmap tasks — #4 and #5. By the end of Step 5 you should be able to mark both as done.
1

Scaffold the module with the Extension Wizard

What

Use Slicer's built-in Extension Wizard to generate a new scripted module skeleton — let's call it NeedlePathPlanner.

Why

A scripted module isn't just "a Python file" — Slicer needs a specific file/folder layout and a specific set of classes with names derived from the module name in order to discover and load it into the Modules dropdown. The Extension Wizard generates exactly that boilerplate correctly, registers the module's path with Slicer, and gives you a working "Hello World" module you can immediately see and run — so from minute one you're editing a real, loadable module instead of debugging plumbing.

How

  1. In the Modules dropdown, search for "Extension Wizard" (under Developer Tools) and switch to it.
  2. Click "Create Extension". Choose a parent folder — put it inside this repo, e.g. D:\MedVis\slicer-module. Give the extension a name, e.g. MedVisExtension.
  3. It will then prompt you to create a module inside that extension. Name it NeedlePathPlanner (CamelCase — this becomes the Python class name) and keep the default "Scripted Loadable Module" template.
  4. The Wizard creates MedVisExtension/NeedlePathPlanner/NeedlePathPlanner.py (plus a Resources/UI/NeedlePathPlanner.ui file and an icon) and automatically adds this path to Slicer's module search paths.
  5. Search the Modules dropdown for "NeedlePathPlanner" — you should see a basic UI with an input/output volume selector and an "Apply" button. That's the generated template running.

Underlying concept

A Slicer Extension is a distributable package (think: a plugin) that can bundle one or more Modules. A scripted module is a Python class inheriting from ScriptedLoadableModule. The generated file actually contains three or four classes:

ClassRole
NeedlePathPlanner Metadata only — title, category, help text, dependencies. Slicer reads this to populate the Modules menu.
NeedlePathPlannerWidget The UI layer — builds/loads the panel, wires up button clicks, reads/writes widget values.
NeedlePathPlannerLogic The actual computation — completely independent of the UI, so it can be called from the Python console, other modules, or tests.
NeedlePathPlannerTest A self-test Slicer can run automatically (Reload & Test button).
Interview angle: the Widget/Logic split is a deliberate separation-of-concerns pattern (similar to MVC/MVP). Be ready to explain why: logic that lives only inside a button's click-handler can't be unit tested, reused from the Python console, or called by another module. Keeping it in a plain Python class with no Qt dependencies makes it independently testable.
2

Build the UI: node selectors and action buttons

What

Edit Resources/UI/NeedlePathPlanner.ui to add:

  • A volume selector (the CT/MR you're planning on)
  • A segmentation selector (the vessel/organ "no-go" structure)
  • Two buttons: "Place Entry Point" and "Place Target Point"
  • A "Plan Trajectory" button
  • A read-only result label

Why

The module needs to know which volume and which segmentation in the scene to operate on — the user might have several loaded. It also needs a way for the user to trigger two distinct interactions (placing two different points) and to see the outcome. Doing this with Slicer's own widget classes means the UI automatically stays in sync with the scene (e.g. if the user loads a new segmentation, it appears in the dropdown immediately) without you writing any refresh code.

How

  1. From the Extension Wizard, use "Edit module UI" (or open the .ui file directly) — this launches Qt Designer with Slicer's custom widgets available in the palette.
  2. Drag a qMRMLNodeComboBox onto the form for the volume selector. In its properties, set nodeTypes to vtkMRMLScalarVolumeNode.
  3. Add a second qMRMLNodeComboBox for the segmentation, with nodeTypes = vtkMRMLSegmentationNode.
  4. Add three QPushButtons ("Place Entry Point", "Place Target Point", "Plan Trajectory") and one QLabel for the result. Give each a clear objectName (e.g. placeEntryButton) — this is the name you'll use in Python as self.ui.placeEntryButton.
  5. Save. Back in Slicer, click "Reload" on the module — your new widgets appear immediately (no restart needed).

Underlying concept

This is your first real contact with MRML (Medical Reality Modeling Language) — Slicer's in-memory data model. Everything loaded into Slicer (volumes, segmentations, fiducials, models, transforms) is a node in the vtkMRMLScene (accessible in Python as slicer.mrmlScene). qMRMLNodeComboBox is a Qt widget that watches the scene and auto-populates with nodes matching a given class filter — a concrete example of the observer pattern VTK/MRML is built on: widgets and logic subscribe to scene-modified events rather than polling.

Interview angle: "How does Slicer represent loaded data, and how does the UI stay in sync with it?" → MRML scene + node hierarchy + observer/callback events. This is the same pattern (subscribe to change events rather than poll) you'll see in any GUI framework, and it's also how the 3D/slice views stay in sync with each other.
3

Place entry/target fiducials and build the trajectory line

What

Wire the two "Place..." buttons to Slicer's interactive point-placement mode, storing each point in a vtkMRMLMarkupsFiducialNode. Wire "Plan Trajectory" to read both point positions and create a line between them.

Why

This is the planning interaction: "entry point" = where the needle enters the skin, "target point" = the lesion/structure to reach. The line between them is the planned trajectory — both the thing the user visualizes and the geometric object the clearance check (Step 4) will test.

How

In NeedlePathPlannerLogic (or the Widget, for the placement calls):

# Create (or get) a fiducial node to hold our two points
fiducialNode = slicer.mrmlScene.GetFirstNodeByName("EntryTarget")
if not fiducialNode:
    fiducialNode = slicer.mrmlScene.AddNewNodeByClass(
        "vtkMRMLMarkupsFiducialNode", "EntryTarget")

# Activate interactive placement (called from the "Place Entry Point" button)
selectionNode = slicer.app.applicationLogic().GetSelectionNode()
selectionNode.SetActivePlaceNodeID(fiducialNode.GetID())
interactionNode = slicer.app.applicationLogic().GetInteractionNode()
interactionNode.SetCurrentInteractionMode(interactionNode.Place)

# Later, reading the two points back (RAS coordinates, in mm)
entryRAS = [0, 0, 0]
targetRAS = [0, 0, 0]
fiducialNode.GetNthControlPointPositionWorld(0, entryRAS)
fiducialNode.GetNthControlPointPositionWorld(1, targetRAS)

# Build the trajectory as a line model so we can run geometry on it later
lineSource = vtk.vtkLineSource()
lineSource.SetPoint1(entryRAS)
lineSource.SetPoint2(targetRAS)
lineSource.Update()

trajectoryModel = slicer.mrmlScene.GetFirstNodeByName("TrajectoryLine")
if not trajectoryModel:
    trajectoryModel = slicer.mrmlScene.AddNewNodeByClass(
        "vtkMRMLModelNode", "TrajectoryLine")
    trajectoryModel.CreateDefaultDisplayNodes()
trajectoryModel.SetAndObservePolyData(lineSource.GetOutput())

Exact method names occasionally shift between Slicer versions — when you implement this, test each call in the Python Console first (instant feedback, autocomplete with Tab), then move working code into the module.

Underlying concept

Two concepts to nail down here:

  • RAS coordinates. Slicer's internal coordinate system is Right, Anterior, Superior — i.e. +X points to the patient's right, +Y to anterior, +Z to superior, in millimeters. DICOM and ITK use LPS (Left, Posterior, Superior) — same axes, two of them flipped in sign. Whenever you exchange coordinates between Slicer and ITK/numpy code (e.g. for the MONAI stretch goal), you need to convert between RAS and LPS or you'll get mirrored results.
  • Markups vs. Models. vtkMRMLMarkupsFiducialNode / ...LineNode are annotation nodes — lightweight, interactive, great for points the user places/drags. A vtkMRMLModelNode wraps actual vtkPolyData geometry — what you need once you want to run geometric algorithms (like an intersection test) rather than just display something.
Interview angle: "Why do medical imaging tools care about coordinate systems like RAS vs LPS?" — because a sign error on one axis silently mirrors left/right or flips superior/inferior, which in a clinical planning tool is exactly the kind of bug that becomes a hazard (see Module 6, ISO 14971) rather than just a visual glitch.
4

Convert the segmentation to a surface and test for intersection

What

Convert the selected vessel/organ vtkMRMLSegmentationNode to a closed surface mesh, then test whether the trajectory line intersects that mesh using vtkOBBTree.

Why

This is the safety-critical step — the system function "Verify clearance" from the Arcadia model (Module 4) and the mitigation for the "vessel puncture" hazard (Module 6) both come down to this one geometric test: does the planned line touch the forbidden region?

How

# 1. Make sure the segmentation has a closed-surface representation
segmentationNode.CreateClosedSurfaceRepresentation()

# 2. Get that surface as vtkPolyData for the first/only segment
segmentIds = vtk.vtkStringArray()
segmentationNode.GetSegmentation().GetSegmentIDs(segmentIds)
segmentId = segmentIds.GetValue(0)

polyData = vtk.vtkPolyData()
segmentationNode.GetClosedSurfaceRepresentation(segmentId, polyData)

# 3. Build a spatial search structure over the surface
obbTree = vtk.vtkOBBTree()
obbTree.SetDataSet(polyData)
obbTree.BuildLocator()

# 4. Test the entry->target line against it
intersectionPoints = vtk.vtkPoints()
hit = obbTree.IntersectWithLine(entryRAS, targetRAS, intersectionPoints, None)

crossesVessel = bool(hit) and intersectionPoints.GetNumberOfPoints() > 0

Underlying concept

Segmentations are stored internally as label maps — a 3D grid of voxels where each voxel is tagged with a segment ID. Most geometric algorithms (intersection, distance, volume) don't operate on voxel grids directly — they need a surface mesh (triangles). Converting label map → closed surface uses a marching cubes-family algorithm — the exact technique Module 7 (VTK/ITK deep dive, Preim's book) has you reproduce by hand, so this step is a preview of that.

vtkOBBTree is a bounding volume hierarchy: it recursively wraps groups of triangles in oriented bounding boxes, so a line-intersection test can discard most of the mesh in O(log n) box checks instead of testing every triangle individually — the same family of acceleration structure used in ray tracing and collision detection.

Interview angle: "How would you efficiently test whether a line intersects a complex 3D mesh with thousands of triangles?" — don't brute-force every triangle; use a spatial acceleration structure (BVH / OBB tree / octree) to prune most of the mesh first. Naming vtkOBBTree and explaining why it's faster than the naive approach is a strong signal.
5

Show the result — and connect it back to the SE artifacts

What

Update the result label based on crossesVessel: green "Clear" if False, red "Crosses vessel — re-plan" if True. Re-run this every time "Plan Trajectory" is clicked.

Why

A correct-but-invisible check is useless to a clinician. This step turns the geometric result from Step 4 into the thing the user actually needs: a clear go/no-go signal, displayed where they're already looking.

How

if crossesVessel:
    self.ui.resultLabel.setText("⚠ Crosses vessel — re-plan")
    self.ui.resultLabel.setStyleSheet("color: #b91c1c; font-weight: bold;")
else:
    self.ui.resultLabel.setText("✓ Clear")
    self.ui.resultLabel.setStyleSheet("color: #15803d; font-weight: bold;")

Underlying concept

This is the point where the "build it" half and the "describe and de-risk it" half of the project become the same artifact viewed two ways:

  • In Capella (Module 4), this is the system function "Verify clearance" in the System Analysis layer, realized by the "Clearance Checker" logical component.
  • In the requirements spec (Module 5), this is a requirement like "REQ-0xx: The system shall indicate whether the planned trajectory intersects the selected no-go segmentation before the user proceeds."
  • In the hazard analysis (Module 6), this is the mitigation for the "vessel puncture" hazard — and the crossesVessel boolean is literally the thing being verified.
Checkpoint — you're done with Module 3 when: the module appears in the Modules dropdown; you can load a volume + a segmentation, click "Place Entry Point" and "Place Target Point" and click in a slice view to set each; click "Plan Trajectory" and see the line appear in the 3D view; and the result label correctly says "Clear" for a line that misses your test segmentation and "Crosses vessel" for one that hits it. That satisfies roadmap Task #4 and Task #5.
Interview angle: if asked "tell me about a project where you connected software design to requirements/risk" — this is it. The traceability isn't a paperwork exercise bolted on afterwards; each SE artifact names a real function/component/check that exists in the code you just wrote.
