Requirements Specification — study version

A standards-annotated walkthrough of the requirements for the NeedlePathPlanner needle-planning module. Companion to the canonical requirements.md / traceability-matrix.md and Module 5 of the learning guide.

Honesty note. This is a learning-scale artifact that emulates the structure of a medical-device software requirements specification. It is not a regulatory deliverable — there is no quality management system behind it, no formal review/approval, and no full risk file. Its purpose is to show you understand how requirements, the standards, and traceability fit together.

1. Why a needle planner needs all these standards

For software that is itself a medical device, sold in the EU / Germany, several standards govern the requirements. Each one touches the spec from a different angle:

EU MDR 2017/745Market access in the EU Demands a defined intended purpose and conformance to the General Safety & Performance Requirements (GSPRs, Annex I). Software-as-a-medical-device is classified via Rule 11.

2. Intended purpose & safety classification

Intended purpose. NeedlePathPlanner is research/educational software intended to assist a trained interventional radiologist in planning a percutaneous needle path on a patient's CT/MR images. Given a chosen entry point and a target lesion (e.g. a tumour), it displays the straight-line trajectory between them and indicates whether that path would intersect a user-designated no-go structure such as a blood vessel — helping the clinician reach the target without rupturing critical vessels. It supports planning for percutaneous surgery, intervention, and diagnostic procedures. It is a planning aid only; the radiologist makes the final decision.

Software safety class (IEC 62304): Class B — a planning aid with a mandatory human-in-the-loop and independent intra-procedure imaging as external risk controls. (A conservative reviewer could argue Class C, since a false "Clear" could contribute to a vessel puncture.)

3. How to read a requirement

Every row is one atomic, verifiable statement using "shall", with a stable ID. Type tags: F functional · NF non-functional (quality) · C constraint. A RC flag means the requirement exists to control a hazard (ISO 14971).

6. Standards reference (for study)

IEC 62304 — medical device software lifecycle

The backbone here. Key ideas: classify software by worst-case harm (A/B/C); perform software requirements analysis (§5.2) — each requirement uniquely identified, verifiable, and covering categories like functional, inputs/outputs, interfaces, and risk control measures (§5.2.2); and maintain bidirectional traceability from system requirement → software requirement → architecture → test.

Be ready to say: "Our tool is Class B because a human-in-the-loop is a mandatory external risk control; I documented the reasoning rather than just asserting a class."

ISO 14971 — risk management

Defines the risk process: identify hazards, estimate/evaluate risk, implement risk controls, and verify them. Every risk control must trace to a requirement — those are our RC rows (e.g. REQ-006 "determine intersection", REQ-009 "no false Clear"). Module 6 builds the hazard table that these link to.

IEC 62366-1 — usability engineering

Targets use error — harm caused not by a software fault but by the user misreading or misusing the interface. REQ-007 (show a result before proceeding), REQ-008 (colour-code it) and REQ-010 (name the tested segment) are use-related requirements: they make the safety-relevant information impossible to miss or misread.

EU MDR 2017/745 — market regulation

Requires a defined intended purpose (section 2 above) and conformance to the General Safety & Performance Requirements (Annex I). Software is classified under Rule 11 — software providing information used for diagnostic/therapeutic decisions is typically Class IIa or higher. Performance claims (e.g. REQ-019) ultimately answer to the GSPRs.

IREB CPRE — requirements quality

Not a regulation but the source of our quality bar: every requirement must be unambiguous, verifiable, atomic, necessary, feasible, consistent. This is why "the system shall quickly and clearly handle the data" is rejected (not measurable, not atomic) and split into precise, testable statements.
