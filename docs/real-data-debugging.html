Debugging Log: Loading Real Decathlon Data into NeedlePathPlanner

A record of swapping the placeholder test segmentation for real Medical Segmentation Decathlon data (Task08_HepaticVessel) — what went wrong, why, and the concepts behind each fix. This is a companion to the Module 3 build walkthrough and Module 2; it picks up once the module already works against the placeholder segmentation.

Why this page exists: the build steps in steps.html show the happy path of building the module. Real datasets don't follow happy paths — they surface assumptions your code quietly made that a single hand-rolled test segmentation never challenged. Each section below is one of those assumptions breaking, and what it teaches.
A

Getting the dataset without filling the disk

What happened

Task08_HepaticVessel.tar is ~8.8 GB. The machine's C: drive had only 5.4 GB free — extracting the whole archive there would have failed partway through, or worse, filled the drive to 100%.

The fix

Two separate ideas solved this:

  • Extract to a drive with room (D:, 329 GB free) instead of the default Downloads location.
  • Don't extract the whole archive. A Decathlon task has hundreds of cases; testing a pipeline only needs one. tar can extract individual entries by path instead of everything: 
    tar -tvf Task08_HepaticVessel.tar | grep imagesTr   # list contents + sizes first, extract nothing
tar -xf Task08_HepaticVessel.tar -C D:\MedVisData Task08_HepaticVessel/imagesTr/hepaticvessel_248.nii.gz Task08_HepaticVessel/labelsTr/hepaticvessel_248.nii.gz

Underlying concept

A .tar file is just a concatenation of files plus headers — it doesn't require reading or writing the whole archive to pull out one entry. This is the same reason you can git checkout a single file from a huge repo's history without materializing every version that ever existed: archive/container formats that store an index let you do partial extraction cheaply.

Interview angle: "How would you work with a dataset too large to fit on disk?" — don't extract/download more than the task needs. Sample a subset, stream what you can, and know the actual disk budget before running a bulk operation.
B

NIfTI, gzip, and the Add Data dialog

What happened

Windows Explorer displayed the extracted file as hepaticvessel_248.nii, which looked like maybe the .gz part had gone missing during extraction. Separately, Slicer's Add Data dialog only offered Volume / Segmentation / Transform as type options — no LabelMap, which earlier guidance had assumed would be there.

The fix / explanation

  • .nii.gz is a NIfTI file (3D medical image format) compressed with gzip. Windows hides the last extension by default ("Hide extensions for known file types") — the file on disk was unaffected, only its on-screen label changed. Slicer decompresses gzip natively on load.
  • Current Slicer versions can read an integer-labeled NIfTI directly into a vtkMRMLSegmentationNode by choosing Segmentation in the Add Data dialog — it auto-creates one segment per distinct label value. This skips the older two-step "load as LabelMap, then import LabelMap → Segmentation" workflow entirely.
Interview angle: "Tell me about a time documentation/instructions were slightly out of date." — tooling evolves faster than tutorials. The fix wasn't to force the old workflow to work; it was to recognize the newer, more direct option achieved the same goal with fewer steps.
C

Duplicate nodes after Add Data

What happened

After loading 2 files (1 volume, 1 segmentation), the scene ended up with 4 nodes: three volume-looking nodes and one segmentation. The Add Data dialog's file list had shown each path with the filename appearing twice (.../hepaticvessel_248.nii/hepaticvessel_248.nii) — a real symptom of a double-registered file, not just a cosmetic glitch.

The fix

Rather than guess which node was "the real one," each ambiguous volume was loaded as the slice-view background and inspected visually: a real CT shows recognizable grayscale anatomy; a mislabeled duplicate of the label file looks mostly black with only 1–2 flat values. Once the real CT was identified, the duplicates were deleted.

Interview angle: "How do you debug a data-loading issue with no error message?" — when nothing throws an exception but the data looks wrong, fall back to direct inspection of the actual values/voxels rather than trusting names or metadata, which can be misleading exactly when something went wrong upstream.
D

Quiz: telling vessel from tumor by shape alone

What happened

Task08's segmentation has two segments — Decathlon's labels are just integers (1, 2); Slicer names them generically Segment_1, Segment_2. Nothing in the UI tells you which one is the vessel and which is the tumor — but their shapes in the 3D view do.

Click the shape that's the vessel

One of these is a real structure shape from the 3D view; the other is what a tumor looks like. Click the one you think is the vessel.

Underlying concept

Vessels physically branch — they're a tree-like network feeding/draining tissue, narrowing at each split. Tumors are solid masses with no branching structure. This shape difference is visible even without any segment naming metadata, and it's exactly how the real bug below was first spotted: the two segments in the actual 3D view had unmistakably different shapes.

E

The real bug: testing against the wrong segment

What happened

checkClearance() picked whichever segment came back first from GetSegmentIDs() — segmentIds.GetValue(0) — with no regard for which segment that actually was. With one hand-made test segment, "segment 0" was always the only segment, so this never mattered. With two real, differently-positioned segments, it suddenly did: every trajectory aimed at the tumor came back "Clear", even ones that looked obviously close to it, because the code was silently testing against the vessel — somewhere else entirely — instead.

Try it yourself: buggy vs. fixed segment selection

The green branching shape is the vessel (Segment_1); the tan blob is the tumor (Segment_2). Place an entry and target point, then plan a trajectory in either mode and see what gets tested.

No trajectory planned yet

The fix

The segmentation node selector alone isn't enough — a node can contain multiple segments with different clinical meanings (one to avoid, one to target). The selection of which segment is the no-go structure needs to be an explicit, visible choice in the UI, not an arbitrary index baked into the logic.

Underlying concept

This is the same class of bug as reading list[0] from an API response and assuming it's "the main one" — it works by coincidence until the data has more than one element in an order you didn't control. The fix is always the same shape: make the selection explicit and visible, rather than implicit and positional.

Interview angle: "Describe a bug that only showed up with real data, not your test data." — single hand-crafted test fixtures often have exactly one of everything, which silently hides bugs in code that assumes "there's only one." Testing against realistic, messier data (multiple segments, multiple structures) is what surfaces this class of bug before a clinician does.
Status: identified, reproduced (interactively, above), and fixed — see Step F below.
F

The fix: an explicit segment selector

What changed

The segmentation node combo box (qMRMLNodeComboBox) was replaced with a qMRMLSegmentSelectorWidget — a single Slicer widget that picks both the segmentation node and a specific segment inside it. The logic now receives that chosen segment ID explicitly: 

# Widget (onPlanTrajectoryButton):
segmentationNode = self.ui.segmentationSelector.currentNode()
segmentId        = self.ui.segmentationSelector.currentSegmentID()
crossesVessel    = self.logic.checkClearance(segmentationNode, segmentId)

# Logic (checkClearance) — was: chosenId = segmentIds.GetValue(0)
if not segmentId or not segmentation.GetSegment(segmentId):
    raise ValueError("No valid segment selected to check the trajectory against")
segmentationNode.GetClosedSurfaceRepresentation(segmentId, polyData)

Why this shape of fix

The selection of which segment is the no-go structure is now an explicit, visible choice in the UI rather than an arbitrary positional index baked into the logic. With the vessel and tumor both present, you now decide which one the trajectory is tested against — and the result can no longer be right or wrong by accident of load order.

Interview angle: "How did you fix it?" — not by reordering the data or guessing which index is the vessel, but by removing the positional assumption entirely: surface the choice to the user and pass it through. The safety property no longer depends on something the code can't control.
To verify in Slicer: Reload the module (or Restart Slicer if the new segment dropdown doesn't appear — the .ui gained a custom widget). Pick the segmentation, then explicitly select the vessel segment. Aim a trajectory through the tumor but clear of the vessel → should now say Clear (correct: you're checking against the vessel). Switch the selector to the tumor segment and re-plan the same line → it should flip to Crosses. The result now tracks the segment you chose.
