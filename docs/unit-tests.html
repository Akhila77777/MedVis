Unit Tests — study version

How the automated tests for the NeedlePathPlanner Slicer module work, and what each one proves. Companion to the code in NeedlePathPlanner.py and to the requirements / risk pages.

1. Why these tests exist

The clearance check is the safety-critical part of this tool — a wrong answer can contribute to a vessel puncture (hazard H-01). IEC 62304 expects software units like this to be verified by testing, not just clicked through by hand. So every test below is also a row of verification evidence in the traceability matrix: it proves a specific requirement is actually met, and it will catch any future change that breaks it (a regression test).

The tests subclass Slicer's ScriptedLoadableModuleTest. The runner calls runTest(), which runs each test after a setUp() that clears the scene — so every test starts from a clean, isolated state.

2. How to run them

In Slicer's NeedlePathPlanner module: Reload & Test → "Reload and Test". Or, for explicit output, paste this one line into the Python console:

import NeedlePathPlanner; slicer.mrmlScene.Clear(); NeedlePathPlanner.NeedlePathPlannerTest().runTest()

No traceback = all 8 passed. A test failure prints a Traceback ending in an AssertionError.

5. The eight tests

test_logicLoads smoke

Confirms the NeedlePathPlannerLogic class instantiates without error. The "is the wiring intact" baseline — if this fails, nothing else matters.

test_planTrajectory_buildsLine REQ-004

Places an entry and target point, calls planTrajectory(), and asserts a TrajectoryLine model now exists with exactly 2 points. Proves the trajectory is actually built between the two points.

test_checkClearance_crossing_returnsTrue REQ-006

Positive test. Entry (-50,0,0) → target (50,0,0) drives a line straight through the cube. Asserts checkClearance() returns True (crosses). This is the core "detects danger" behaviour.

test_checkClearance_clear_returnsFalse REQ-006

The mirror case: a line 100 mm above the cube. Asserts False (clear). Together with the previous test it proves the check is discriminating — it says yes when it should and no when it should, not just one of them.

test_checkClearance_invalidSegment_raises REQ-005 / REQ-009

Negative test. Passes a segment id that doesn't exist. Asserts a ValueError is raised. This is the test that guards against the original GetValue(0) bug: the code must refuse to answer rather than silently test the wrong thing and return a false "Clear".

with self.assertRaises(ValueError):
    NeedlePathPlannerLogic().checkClearance(segmentationNode, "no-such-segment")

test_planTrajectory_missingTarget_raises REQ-014

Negative test. Only an entry point is placed. Asserts planTrajectory() raises rather than proceeding with incomplete input — the tool must not act on half a plan (hazard H-03).

test_checkClearance_emptySegmentation_raises REQ-018

Negative test. A segmentation node with no segments. Asserts a ValueError instead of a crash or a blank result, so the clinician can never proceed on a check that never ran (hazard H-03).

test_checkClearance_recomputesAfterReplan REQ-017

Regression test. Checks a crossing config returns True, then moves the points to a clear config and asserts it now returns False. This proves the result follows the live points and never goes stale (hazard H-02 — trusting an out-of-date "Clear").

# crossing -> True
self._setSinglePoint("EntryPoint", [-50,0,0]); self._setSinglePoint("TargetPoint", [50,0,0])
self.assertTrue(logic.checkClearance(segmentationNode, segmentId))
# move clear -> must now be False (recomputed, not cached)
self._setSinglePoint("EntryPoint", [-50,100,0]); self._setSinglePoint("TargetPoint", [50,100,0])
self.assertFalse(logic.checkClearance(segmentationNode, segmentId))

6. What's covered — and what isn't (honest)

Unit-tested: REQ-004, REQ-005, REQ-006, REQ-009, REQ-014, REQ-017, REQ-018.

Still manual-only (honestly): REQ-015 multi-segment selection (needs real multi-segment data, not a single cube), REQ-019 performance timing, and the visual/usability requirements (REQ-007/008/010). These are documented as gaps rather than hidden — which is the point of the exercise.
Interview angle. "I have positive tests for the correct answer, negative tests proving bad input is rejected rather than silently mishandled, and a regression test that the result can't go stale — each traced to a requirement and a hazard." That sentence demonstrates real verification thinking.
